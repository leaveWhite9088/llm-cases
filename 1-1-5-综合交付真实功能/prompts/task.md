# 综合前端交付：Signal Loom 客户证据工作台

请从零交付一个可以真正使用的客户研究工作台，而不是静态概念页。

Signal Loom 面向 5–20 人的产品团队。研究员把访谈、支持工单和销售记录中的证据整理成“信号”，产品负责人据此判断哪些问题值得进入路线图。页面的核心工作是：让团队快速捕捉证据、建立主题、判断证据强度，并能安全保存和迁移数据。

## 技术约束

- 使用原生 HTML、CSS 和 JavaScript，不使用框架、构建工具、CDN、网络字体或远程资源。
- 页面必须通过 `index.html` 直接运行，至少交付 `index.html`、`styles.css`、`script.js`。
- 所有数据和资源均离线可用；禁止 base64 大图和复制现成网站。
- 需要提供不少于 8 条真实、具体的初始信号数据，不使用 lorem ipsum。

## 产品功能

### 1. 工作台概览

- 品牌 `Signal Loom`，页面主标题 `Evidence, woven into decisions.`。
- 展示总信号数、强证据数、待处理数，以及按主题聚合的简洁可视化。
- 提供清楚的主操作 `Capture signal`。

### 2. 信号库

- 每条信号至少包含：唯一 ID、标题、原始摘录、来源（Interview / Support / Sales）、主题、证据强度（1–3）、状态（Inbox / Synthesized / Planned）、记录日期。
- 支持全文搜索、来源筛选、主题筛选、状态筛选和排序。
- 筛选条件可以组合，并显示当前结果数量；无结果时提供清除筛选动作。
- 点击信号打开详情面板，完整展示信息并提供编辑、删除和状态推进操作。

### 3. 新建、编辑和删除

- `Capture signal` 打开可访问的表单对话框。
- 必填字段有明确标签和内联校验；错误后保留用户已经输入的内容。
- 新建和编辑后列表、指标、主题汇总立即同步。
- 删除前确认；删除后显示可操作 Toast，并在至少 8 秒内允许 `Undo`。
- 所有新建 ID 必须稳定且不与现有数据重复。

### 4. 本地持久化与数据迁移

- 使用版本化 localStorage 键 `signal-loom:v1`；刷新后恢复数据。
- localStorage 丢失、JSON 损坏或结构错误时安全回退到种子数据，并给出非阻断提示。
- 提供 `Export JSON`，下载内容应包含 `schemaVersion: 1` 和全部信号。
- 提供 `Import JSON` 文件选择；导入前验证版本、数组结构、必填字段、枚举值和重复 ID。
- 非法文件不得覆盖现有数据，要显示具体错误；合法导入后更新界面并持久化。

### 5. 响应式与产品体验

- 1440px 桌面端呈现高效的研究工作台；390px 手机端保持完整任务闭环且无横向滚动。
- 移动端筛选可以折叠，但当前筛选状态必须可见。
- 对话框/详情面板需要合理管理焦点，Escape 可关闭，关闭后焦点返回触发控件。
- 所有按钮具备 hover、active、focus 和 disabled 状态。
- 支持 `prefers-reduced-motion`，颜色不能作为状态的唯一表达方式。

## 视觉方向

不要做通用 SaaS 仪表盘。把界面设计成“研究材料编织台”：冷灰蓝工作底板、清楚的资料标签、细线构成的证据脉络，以及一处醒目的信号橙色用于关键动作。界面应有编辑部工具的精确感，而不是堆叠渐变和玻璃卡片。

## 验收前自查

至少自行走通：初始加载、组合筛选、新建、编辑、删除与撤销、刷新恢复、合法导入、非法导入、空结果、键盘操作、桌面与手机视口。直接实现，不要只输出设计方案或代码片段。
