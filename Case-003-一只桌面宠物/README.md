# Case-003 一只桌面宠物

> 一只住在浏览器角落的小动物，从会动到有性格，一次交付

**你的任务：** 按本文档要求，在合适的实例目录下完成桌面宠物的实现。

工作目录与实例编号见下一节——它决定你在哪个目录里写代码。

## 场景

在浏览器角落养一只像素风格的小动物（默认形象：一只圆滚滚的小猫）。整个 Case 是一次性交付——你需要在一个工作会话里把整只「能互动、能记住状态、能跨标签页跟着你」的宠物做完。

## 必含元素

1. **角色视觉**：圆滚滚的像素小猫，暖色调，自包含资源（不依赖外部图片文件）
2. **基础动作**：待机、走路、转鼠标（看向鼠标方向）至少三个
3. **状态系统**：饥饿、困倦、心情等状态，状态影响动画表现（饿了走路变慢、困了打哈欠、被摸会开心）
4. **互动**：点击 / 拖动有反馈（拖动改变位置，点击喂食 / 摸头）
5. **持久化**：状态保存到 localStorage，下次打开恢复（包括位置、状态值）
6. **多标签页同步**：用 BroadcastChannel 或 storage 事件，多标签页是同一只猫（状态变化、动作同步）

## 技术约束

- 角色资源自包含：纯 CSS / SVG / Canvas 实现，不依赖外部图片文件
- 嵌入方式：Web Component 或浮窗页面（能嵌入任意页面）
- 在 Chrome / Edge 最新版流畅运行
- 渲染方式任选 CSS / SVG / Canvas，但要保证可爱感
- 单文件 HTML 交付，所有代码封装在一个 `index.html` 中

## 不需要做的

- 真实桌面应用（Electron 等），浏览器内即可
- 复杂养成系统（数值、等级），简单状态即可
- 移动端适配，桌面浏览器跑通即可

## 动画原理

桌面宠物的"灵魂"在于动作的自然感与状态变化的可感知度。这三点决定了它是否"看起来是活的"。

1. **呼吸感**：待机不是死板的不动，而是有微小的呼吸、眨眼、耳朵抽动等细微动作。这些微动作是"活着"的标志——没有它们，再漂亮的猫也像截图。

2. **状态驱动动画**：饥饿时走路变慢、表情变差；困倦时打哈欠、眯眼、动作变少；心情好时被摸会撒娇。状态变化必须用动画表现，不是用数字或图标——用户看不到 localStorage 里的数字，他们只看到猫。

3. **互动即时性**：点击 / 拖动要立刻有反应（不是 100ms 后才反应）。反应可以是动作（跳一下、蹭手）、表情（眯眼）、声音（短促音效任选其一）。

请确保：
- 待机有微动作（不是静止帧）
- 状态变化有可见的动画表现（不是只有数字变）
- 互动反应即时且明确

## 场景设置

### 栖息环境
- 默认在浏览器右下角浮窗显示
- 可拖动到屏幕任意位置
- 多个标签页中是同一只猫，位置与状态同步

### 角色
- 形象：圆滚滚的像素小猫，暖色调
- 尺寸：约 64×64 到 128×128 像素（不占太多屏幕，但能看到细节）
- 资源自包含：纯 CSS / SVG / Canvas 实现（不依赖外部图片文件）
- 风格方向任选：
  - **像素艺术**：8-bit / 16-bit 风格的方块拼贴
  - **柔和矢量**：圆角几何 SVG，扁平但温暖
  - **手绘风**：用 SVG 路径模拟手绘笔触

### 状态表达
- **饥饿**：走路变慢 + 表情变化 + 偶尔看向屏幕外的"食物"方向
- **困倦**：打哈欠 + 眯眼 + 整体动作幅度变小
- **心情好**（被摸后）：蹭手 + 眯眼笑 + 偶尔跳一下
- **心情一般**（被忽略一段时间）：尾巴低垂 + 偶尔叹气

## 动画流程

### 阶段 1：进入与待机（视觉高潮 #1，开屏冲击）
页面打开时，宠物出现并进入待机。

**视觉表现：**
- 入场动画：从无到有（淡入 / 缩小出现 / 跳跃出场，挑一种）
- 进入后进入待机状态：呼吸（缩放微动）、偶尔眨眼、耳朵随声音抽动
- 鼠标移近时转头看向鼠标方向（不是瞬转，是平滑过渡）

**关键质量点：**
- 第一眼的"可爱冲击"决定用户愿不愿意继续玩下去
- 待机必须有微动作，不然像贴图
- 鼠标跟随必须流畅，不能一抖一抖

### 阶段 2：互动反应（视觉高潮 #2，被摸的瞬间）
用户点击、拖动、长按时。

**视觉表现：**
- **点击**：跳一下 + 眯眼 + 表情变化（如果心情好就撒娇）
- **拖动**：身体跟随鼠标移动，可以是悬浮跟随（身体倾斜像被提着）或拖行
- **放下**：落地时轻微弹跳 / 灰尘
- **长按**：特殊互动——可以设计成喂食（出现食物 icon）或反复抚摸（每次抚摸有不同反应）
- **被摸后**：心情值上升，进入"撒娇模式"数秒（蹭手、眯眼笑）

**关键质量点：**
- 反应即时，不能有明显延迟
- 反馈要"暖"——不是简单的位移，要有情绪表现
- 长按和单击要有区别，不能让用户分不清

### 阶段 3：跨标签页瞬间（视觉高潮 #3，一只猫在多个窗口）
打开第二个标签页时，一只猫同时出现。

**视觉表现：**
- 第二个标签页打开时，宠物以合适的方式出现（淡入 / 从边缘滑入），保持与第一个标签页相同的状态
- 一个标签页里被摸，另一个标签页里能看到"摸完"的余韵（表情短暂同步、或延迟同步）
- 在 A 标签页拖到右边放开，在 B 标签页打开时，宠物从右边滑入（位置同步的方向感）
- 多个标签页的"同一只猫"应该让用户感到惊喜，而不是混乱

**关键质量点：**
- 同步延迟 < 100ms（用户感觉是即时的）
- 跨标签页的位置变化要有"动"的过程，不能瞬移
- 状态变化（饥饿、心情等）必须在所有标签页同步

### 阶段 4：持久化与恢复
关闭浏览器再打开。

**视觉表现：**
- 宠物记得最后位置（拖到屏幕中间就出现在中间）
- 记得所有状态值（饥饿度、心情、上次互动时间）
- 如果距离上次互动很久，状态应该自然衰减（更饿、更困）
- 入场时可以直接进入待机，不需要"恢复中"的过渡

## 视觉与渲染要求

### 角色渲染
- 暖色调（橙、黄、奶白等温暖色系，避免冷色调）
- 自包含资源（不引用任何外部 .png / .jpg / .svg 文件）
- 表情和动作通过资源切换或 transform / opacity 变换实现
- 至少支持：眼睛开 / 闭、嘴开 / 闭、身体倾斜三种状态组合

### 动画性能
- 所有动画优先用 `transform` 和 `opacity`（避免触发 layout / paint）
- 闲置时几乎不消耗 CPU（不要每帧都重绘，待机微动可以用 CSS 动画或 requestAnimationFrame 节流）
- 状态切换有平滑过渡（200-300ms 的渐变，不是硬切）

### 浏览器集成
- 嵌入方式：Web Component（`<my-pet></my-pet>`）或浮窗 iframe
- 多标签页同步：优先用 `BroadcastChannel`，降级到 `storage` 事件
- localStorage 保存：状态值、位置、上次互动时间
- z-index 要高于页面内容但低于模态框（不能挡住关键操作）

## 交互控制

- **点击宠物**：触发短促反应（跳一下 / 表情变化）
- **拖动宠物**：可以拖到屏幕任意位置（位置存 localStorage）
- **长按**：进入特殊互动（喂食 / 持续抚摸，至少一种）
- **鼠标移近**：宠物转头看向鼠标方向
- **右键**（可选）：弹出菜单（喂食 / 玩耍 / 重置）

## 性能要求

- Chrome / Edge 最新版流畅运行（动画 60fps，闲置时不消耗 CPU）
- 多个标签页同时运行（3-5 个）不卡顿
- localStorage 读写不要阻塞主线程
- 角色资源渲染不要触发明显卡顿（避免大量 DOM 节点 / 复杂 SVG 路径）

## 代码要求

- 所有代码封装在一个 HTML 文件中（单文件交付）
- 使用合理的类/模块划分（角色、状态、动画、互动、存储、跨标签页通信）
- 自包含资源（CSS / SVG / Canvas 内部生成角色）
- 不引入框架（纯 HTML / CSS / JS）
- 状态机用清晰的数据结构（避免散落的 if-else）
