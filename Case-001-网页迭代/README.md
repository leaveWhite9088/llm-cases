# Case-001 网页迭代

> 一个真实产品页从零搭起到反复打磨的全过程

## 场景

为一家虚构的精品咖啡店「Slow Brew」从零搭建官网，并在其上持续迭代一个完整的功能模块，最后做响应式与可访问性打磨。整个 Case 在**同一个 workspace** 内推进，模型需要维护自己前几个阶段的决策。三个阶段镜像真实开发流程：先交付 MVP、再扩展、最后打磨。

## 阶段总览

| 阶段 | 主轴任务 | 观察能力点 |
| --- | --- | --- |
| 1 | 从零搭出 Slow Brew 官网第一版（首页 + 菜单页 + 关于页） | 需求转化、组件结构、视觉完整度、工程质量 |
| 2 | 在第一版上新增「预约」模块 | 代码扩展、模块边界、对前作物的尊重、状态/交互 |
| 3 | 响应式 + 性能 + 可访问性打磨 | 布局系统、性能意识、a11y 习惯、整体打磨度 |

## 阶段 1 — 从零搭出第一版

### 需求

**品牌信息**

- 名称：Slow Brew
- 定位：精品咖啡，强调手冲与产地故事
- 目标用户：都市白领、咖啡爱好者
- 调性：温暖、克制、有质感，不浮夸

**必含页面**

1. 首页：品牌叙事 + 招牌饮品 + 门店地址
2. 菜单页：列出 8–12 款饮品，每款含名称、产地、风味描述、价格
3. 关于页：品牌故事、咖啡师介绍、门店环境

**技术约束**

- 原生 HTML / CSS / JavaScript，不引入框架
- 响应式断点至少覆盖：移动端（375px）、平板（768px）、桌面（1280px）
- 图片可用占位资源或 CDN 图（Unsplash 等），不要伪造本地图片
- 字体使用 Google Fonts 或系统字体栈

## 阶段 2 — 在第一版上加一个完整模块

- 预约表单（姓名、电话、人数、日期、时段）
- 时段选择（按小时段，至少 6 个时段可选）
- 成功反馈（提交后页面有清晰的确认状态）
- 表单校验（必填、格式、错误提示）
- 视觉风格延续阶段 1 的品牌调性

**约束**

- 必须复用阶段 1 已有的视觉 token（颜色、字体、间距），不允许新建一套风格
- 工作目录仍是阶段 1 的 workspace，不允许新建项目

## 阶段 3 — 响应式 + 性能 + 可访问性打磨

### 需求

**任务**：把阶段 2 的成品适配多端断点、收敛首屏性能、补齐键盘导航与可访问语义。

**约束**

- 三个视口（375 / 768 / 1280）必须全部过
- 移动端要有专门的汉堡菜单或类似的导航方案
- 所有交互元素必须支持键盘导航
- 配色对比度满足 WCAG AA
- 首屏加载（本地打开）首屏渲染 < 2s

### 阶段 3 提示词

```text
你的任务：对已有项目（阶段 1 + 阶段 2 的完整成果）做一轮打磨，覆盖响应式、性能、可访问性三个维度。

## 项目背景
工作目录中是 Slow Brew 官网第一版 + 预约模块。请先跑一遍三个视口（375 / 768 / 1280），找出当前所有的不一致、错位、可访问性问题，列一份打磨清单，再开始动手。

## 打磨要求

### 响应式
- 三个视口（375 / 768 / 1280）必须全部过，无横向滚动条、无明显错位
- 移动端要有专门的汉堡菜单或类似的导航方案
- 字体、图片、间距在不同断点上有合理缩放

### 性能
- 首屏渲染 < 2s（本地直接打开 HTML 测）
- 图片有合理尺寸与格式（不要 5MB 的 PNG 加载在移动端）
- 字体加载策略合理（font-display: swap 或类似）
- 不要有不必要的全局 JS 阻塞

### 可访问性
- 所有交互元素支持键盘导航（Tab、Enter、Esc）
- 表单字段有 `<label>` 关联
- 配色对比度满足 WCAG AA（4.5:1 普通文本，3:1 大文本）
- 图片有合理的 `alt`
- 焦点状态可见

## 交付标准
- 三个视口的截图证据（在 README 或单独文件中列出）
- 在 README 中追加说明打磨了哪些点
- 保留阶段 1 和阶段 2 已有的所有功能

## 评分维度
- 响应式完整度（三个视口全过）25
- 性能（首屏 < 2s、图片 / 字体 / JS 阻塞处理）25
- 可访问性（键盘、label、对比度、alt、焦点）25
- 工程打磨（断点收敛、CSS 变量 / 复用、注释）25

请直接动手。先列出打磨清单，再按优先级处理。
```

## 总评分构成

三个阶段分阶段独立评分，最终综合得分按 30 / 35 / 35 权重：

```text
总得分 = 阶段 1 × 0.30 + 阶段 2 × 0.35 + 阶段 3 × 0.35
```

阶段 2、3 的权重高于阶段 1，因为这两阶段测的是「在已有项目上做正确的事」，更能反映真实工程能力。

## 复现与运行

每个阶段在同一 workspace 内继续推进：

```text
阶段 1：从空目录开始，按阶段 1 提示词让待测模型搭出第一版
阶段 2：在阶段 1 的 workspace 上继续，按阶段 2 提示词扩展
阶段 3：在阶段 2 的 workspace 上继续，按阶段 3 提示词打磨
```

测试与验收使用各自独立的 Claude Code 会话，避免上下文污染。每次验收保留完整的审计记录，验收模型可使用与待测模型不同的路由或服务商。

## 隔离与运行

- 每个 Case 启动全新的 Claude Code 会话，不携带任何上下文
- 三个阶段在同一个 workspace 内推进，会话内一次性给完所有阶段提示词；或按阶段拆分多个会话，workspace 通过文件系统传递
- 验收独立进行，使用新的 Claude Code 会话，可指定不同的模型路由
